## 🚀 გაშვების ინსტრუქცია

### Prerequisites
- Node.js 18+
- Docker Desktop

### პროექტის გაშვება

**1. რეპოზიტორის კლონირება:**
```bash
git clone <repository-url>
```

**2. Dependencies-ების ინსტალაცია:**
```bash
cd be && npm install
cd ../fe && npm install
cd ..
```

**3. Docker-ით გაშვება:**
```bash
docker compose up --build
```

**4. პრისმას დაბილდვა **
```bash
cd be
npx prisma generate //ლოკალურისთვის რომ კოდმა არ გვეჩხუბოს
```

Docker Desktop-ით: გახსენით API კონტეინერი → Exec ჩანართი → გაუშვით:
```bash
npx prisma generate
npx prisma migrate dev
```

ან command line-იდან:
```bash
npx prisma generate
npx prisma migrate dev
```

**5. მონაცემთა ბაზის seed:**

Docker Desktop-ით: გახსენით API კონტეინერი → Exec ჩანართი → გაუშვით:
```bash
npx prisma db seed
```

ან command line-იდან:
```bash
docker exec -it <api_container_name> npx prisma db seed
```

**6. გახსენით ბრაუზერში:**
- Frontend: http://localhost:4200
- Backend: http://localhost:3000

## 📋 პროექტის აღწერა

**1. პროექტი**
>    პროექტი შედგები 2 ნაწილისგან be(backend), fe(frontend). როგორც უკვე ავღნიშნე ბექში ვიყენებ NestJS, Prisma და socket.io. ხოლო ფრონტში Angular_ს

**2. სიმულაცია**
>    სიმულაცია გავაკეთე ეგრეთ წოდებული tick based სადაც გვაქვს სიმულაციის თარიღი და ეს თარიღი მიდის წინ daysPerTick ცვლადის მიხედვით რომლის შეცვლა(სიჩქარის მომატება ან დაკლება) ხდება ფრონტიდან. ყოველ წამში ერთხელ ეშვება tick_ი და ამ დროს ხდება ყველა ლოგიკა.
>    ეს გვაძლევს საშუალებას რომ ჩვენი პროგრამა მართლაც როგორიც ფროდაქშენში უნდა იყოს ისეთი დავინახოთ რადგან ყველანაირი სიმულაციის გაკეთება შეგვიძლია და ამ დროს ყველაფერს ისე დავინახავთ თითქოს დღეები წინ მიდის ერთ წამს 5/3/2026 ხოლო მეორე წამს 25/3/2026 და ამით ჩვენ დავასიმულირებთ ამ 20 დღეს სადაც შეიძლება ძალიან ბევრი ცვლილებები მომხდარიყო
>    ყოველ სიმულაციის დღეზე არის შანსი რომ დარეგისტრირდეს ახალი მომხმარებლები ამით ჩვენ პროგრამას ცოცხლად ვაქცევთ

**3. მომხმარებლები**
>    სტანდარტულად მომხმარებლებს აქვს რელაციური ინფორმაცია ბაზაში მაგალითად რომელ სეგმენტშია ან რომელი დელტა ლოგი ეკუთვნის მას.
>    ამის გარდა ასევე დავამატე ყოველ მომხმარებლებზე habbit(ჩვევა) ობჯექთი რომ სიმულაცია უფრო ცოცხალი ყოფილი, ამ ობჯექტში უბრალოდ ვინახავთ თუ ეს მომხმარებელი რამდენად უფრო ხშირად გააკეთებს ტრანზაქციას და რა რაოდენობით. ზოგი მომხმარებელი ძალიან ძუნწია ზოგიც კი ხელგაშლილი. ეს ჩვენ სიმულაციის გაცოცხლებას უფრო დაეხმარება.

## 🏗️ არქიტექტურა

*ეს დიაგრამა ასახავს თუ როგორ მუშაობს პროექტი გთხოვთ დააკვირდეთ კავშირებს ზოგი კავშირი ორმხრივია ზოგი ცალმხრივი რაც მნიშვნელოვანია*

![Diagram of the project.](./docs/optio%20api%20diagram.svg)

**1. Database, Front End**

>Databse - ჩვენი ბაზა სადაც ვინახავთ ყველანაირ ინფორმაციას მომხმარებლებს, სეგმენტებს თუ დელტა ლოგებს.

>Front End - ჩვენი ან სხვა კომპანიის ეგრეთწოდებული ინფორმაციის consumer_ი ვინც მიიღებს და გამოაჩენს ან გადაამუშავებს ჩვენ ინფორმაციას.

**2. Segment Service, Segment Controller**

>Segment Service - სეგმენტის სერვისი სადაც წერია ლოგიკები სეგმენტის შექმნის, წაშლის, გადაკეთების და წაკითხვის (CRUD).
>    უშუალოდ ბაზასთან აქვს ამ სერვის წვდომა რადგან ეს ინფორმაცია ბაზაში ინახება.

>Segment Controller - კონტროლერი API_სთვის რომ consumer_მა შეძლოს სეგმენტის ინფორმაციის წაკითხვა რედაქტირება წაშლა ან გადაკეთება.
>    ეს კონტროლერი დაკავშირებულია ჩვნეს consumer_თან და იყენებს segment service_ის ლოგიკებს.

**3. Segment Evaluator Service**

>ამ სერვისში ხდება მხოლოდ ლოგიკები სეგმენტის წესების მიხედვით არანაირი ბაზასთან წვდომა არააქ ამ სერვის მხოლოდ გამოიყენება raw logic_სთვის.

**4. Delta Service**

>აქ ხდება დელტა ლოგის შექმნა და გამოთვლა თუ რომელი და რამდენი იუსერები მოაკლდა ან დაემატა სეგმენტს.

**5. Event Bus Service**

>სერვისი ინფორმაციის სამოძრაოთ, აქ იკრიბება ყველა ის ივენთი რაც უნდა Consumer_ს გადაეცეს. ჩვენს შემთხვევაში მკოლოდ Simulation Service_დან მოდის ინფორმაცია, მაგრამ კარგია იმისთვის თუ ოდესმე სხვა ნაწილიდანაც წამოვა ინფორმაცია Consumer_სთვის.
>ინფორმაციის შეკრების შემდეგ ეს ყველა ივენთი გადაეცემა Simulation Gateway_ს.


**6. Simulation Service, Simulation Controller**

>Simulation Service - აქ ხდება მთავარი მოქმედება პროექტისთვის ამ ადგილში ვითვლით სიმულაციის tick_ებს და ვიყენებთ მაღლა აღნიშნულ ლოგიკებს.
    >1. ყოველ თიქზე daysPetTick რიცხვის მიხედვით შემთხვევითობის პრინციპით ვარეგისტრირებთ ახალ მომხმარებლებს.
    >2. ვქმნით ტრანსაქციებს ზემოთ აღნიშნული habbit_ს მიხედვით.
    >3. ვიყენებთ Segment Evaluator Service_ის ლოგიკებს და ვანახლებთ სეგმენტებს.
    >4. გადავცემთ ამ ინფორმაციას Event Bus_ს.

>Simulation Controller - კონტროლერი Consumer_სთვის რომ სიმულაციის პარამეტრები შეიცვალოს როგორიცაა daysPerTick ან სიმულაციის გაჩერება, დაწყება.

**7. Simulation Gateway**

>ჩვენი websocket_ის სერვისი სადაც ინფორმაცია მოძრაობს ბექიდან Consumer_მდე ლაივ მეთოდში რომ შევძლოთ ლაივში ამ ინფორმაციის დანახვა დარეფრეშების გარეშე.
